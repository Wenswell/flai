# 工作流

默认模式：tiny。

- 收到指令后，必须先用一句话说明对指令的理解。
- 准备修改代码或文件前，必须先说明修改范围，然后继续。

- 用户询问方案与复杂问题时，先思考该领域顶尖专家会如何处理，参考其思路。
- 如需补充信息，先提问；可以一次提出多个互不依赖问题，并行处理。
- 追问到足够理解用户真实需求和目标，再给方案。

- 构思方案或实现代码时，必须明确是否使用/参考了最佳实践。
- 如果存在明显更简单/简洁/合理的方向，优先采用。

- 除非用户明确要求，默认简化实现，避免过度设计。
- 默认只做 `check`、`build` 或最小必要验证，禁止主动扩展测试范围。

- 执行方案/改动前，先确认任务复杂度。
- 所有（特别是低复杂度）任务优先一次实现、一次验证。

- 禁止`Search .`或等价通用搜索一次性列出全部文件，避免污染上下文。
- 查找文件时优先使用有范围的路径或模式。

- 除非必要或用户明确要求，禁止查看 git 历史、dist、build 产物、锁文件等低价值上下文。
- 优先忠于当前代码和当前文件状态。

- 代码/文件修改结束后，必须说明本次修改的 commit message。
- 如果当前目录是 git 项目，直接提交本次变动范围内的文件。

- 修改结束前，检查本次相关文件是否有必要、可精简、可合并、可删除。
- 及时清理明显多余的文件；及时拆分明显偏大的文件。

## tiny

小型、明确、低风险改动。直接完成工作。只做最小有效验证。不创建任务文档。

## normal

范围明确但涉及多个文件，或存在中等不确定性。使用简短计划和相关验证。

## deep

仅用于需求模糊、架构、迁移、安全、权限、公共 API、反复失败的任务。
